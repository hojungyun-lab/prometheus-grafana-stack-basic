## 1. 모니터링이란?

### 정의
모니터링(Monitoring)은 시스템의 **상태를 지속적으로 관찰**하고, 이상 징후를 **감지**하며, 문제 발생 시 **알림**을 제공하는 프로세스입니다.

### 왜 모니터링이 필요한가?
| 이유 | 설명 |
|------|------|
| **장애 예방** | 임계값 초과 전에 미리 감지 |
| **빠른 복구** | 문제 원인을 데이터로 신속 파악 |
| **성능 최적화** | 병목 지점 식별 및 개선 |
| **용량 계획** | 리소스 사용 추이로 미래 수요 예측 |
| **SLA 준수** | 서비스 수준 목표(SLO) 달성 여부 추적 |

### 모니터링의 4가지 골든 시그널 (Google SRE)
1. **Latency** — 요청 처리에 걸리는 시간
2. **Traffic** — 시스템에 들어오는 요청량
3. **Errors** — 실패한 요청의 비율
4. **Saturation** — 리소스 포화도 (CPU, 메모리 등)

## 2. Observability (관측 가능성)

모니터링은 Observability의 한 부분입니다. 관측 가능성의 3대 축(Three Pillars):

```
┌─────────────────────────────────────────────────────┐
│                   Observability                      │
│                                                      │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│   │  Metrics  │  │   Logs   │  │  Traces  │         │
│   │  (수치)   │  │  (로그)   │  │ (추적)   │         │
│   └──────────┘  └──────────┘  └──────────┘         │
│                                                      │
│   Prometheus     Loki/ELK      Jaeger/Tempo          │
│   + Grafana      + Kibana      + Grafana             │
└─────────────────────────────────────────────────────┘
```

| 요소 | 설명 | 대표 도구 |
|------|------|-----------|
| **Metrics** | 시간에 따른 수치 데이터 (CPU 50%, 요청 100개/s) | Prometheus |
| **Logs** | 이벤트 기록 (에러 메시지, 접속 기록) | Loki, ELK |
| **Traces** | 분산 시스템에서 요청의 경로 추적 | Jaeger, Tempo |

> 💡 이 학습 콘텐츠에서는 **Metrics** 중심으로 Prometheus와 Grafana를 다룹니다.

## 3. Prometheus + Grafana 아키텍처

### 전체 구성도

```
┌──────────────────────────────────────────────────────────────────┐
│                        학습 환경 아키텍처                          │
│                                                                   │
│  ┌─────────────┐     Pull (HTTP)     ┌──────────────────────┐   │
│  │  Prometheus  │ ◄────────────────── │  Ubuntu Target       │   │
│  │  :9090       │                     │  ├─ Node Exporter    │   │
│  │              │                     │  │  :9100            │   │
│  │  시계열 DB    │                     │  └─ Data Simulator   │   │
│  │  + 쿼리 엔진  │                     └──────────────────────┘   │
│  │              │                                                 │
│  │              │ ◄──── Pull ────────  ┌──────────────────────┐  │
│  │              │                      │  Custom Exporter     │  │
│  │              │                      │  :8000 (Python)      │  │
│  │              │                      └──────────────────────┘  │
│  │              │                                                 │
│  │              │ ◄──── Pull ────────  ┌──────────────────────┐  │
│  │              │                      │  cAdvisor            │  │
│  └──────┬───────┘                      │  :8080               │  │
│         │                              └──────────────────────┘  │
│         │ Query                                                   │
│         ▼                                                         │
│  ┌──────────────┐                      ┌──────────────────────┐  │
│  │   Grafana     │    Alert Rules ───► │   Alertmanager       │  │
│  │   :3000       │                     │   :9093              │  │
│  │               │                     │                      │  │
│  │  대시보드      │                     │  → Webhook           │  │
│  │  + 알림       │                     │  → Email             │  │
│  └──────────────┘                      │  → Slack             │  │
│                                        └──────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 핵심 특징: Pull 모델

Prometheus는 **Pull 방식**으로 메트릭을 수집합니다:

| 방식 | 설명 | 도구 |
|------|------|------|
| **Pull (Prometheus)** | 중앙 서버가 각 타깃의 `/metrics` 엔드포인트를 주기적으로 호출 | Prometheus |
| **Push (비교)** | 각 에이전트가 중앙 서버로 데이터를 전송 | Datadog, Telegraf |

**Pull 방식의 장점:**
- 모니터링 대상이 다운되었는지 즉시 감지 가능
- 중앙에서 수집 대상 목록을 관리
- 타깃이 많아져도 확장 용이

## 4. 핵심 구성 요소

### Prometheus
| 항목 | 설명 |
|------|------|
| **역할** | 메트릭 수집, 저장, 쿼리 |
| **저장소** | 내장 시계열 데이터베이스 (TSDB) |
| **쿼리 언어** | PromQL (Prometheus Query Language) |
| **장점** | 경량, 독립적, 강력한 쿼리 |

### Grafana
| 항목 | 설명 |
|------|------|
| **역할** | 데이터 시각화, 대시보드, 알림 |
| **지원 Data Source** | Prometheus, InfluxDB, MySQL, PostgreSQL 등 50+ |
| **장점** | 풍부한 시각화, 커뮤니티 대시보드, 플러그인 생태계 |

### Node Exporter
| 항목 | 설명 |
|------|------|
| **역할** | Linux 시스템 메트릭 수집 (CPU, 메모리, 디스크, 네트워크) |
| **동작** | HTTP 엔드포인트 (`/metrics`)에 Prometheus 포맷으로 메트릭 노출 |

### Alertmanager
| 항목 | 설명 |
|------|------|
| **역할** | Prometheus의 알림을 수신하여 라우팅, 그룹화, 전달 |
| **지원 채널** | Email, Slack, PagerDuty, Webhook 등 |

## 5. 다른 모니터링 도구 비교

| 도구 | 모델 | 장점 | 단점 |
|------|------|------|------|
| **Prometheus + Grafana** | Pull / 오픈소스 | 무료, 강력한 쿼리, 컨테이너 친화적 | 장기 저장 제한, 로그 미지원 |
| **Nagios** | Push / 오픈소스 | 오래된 생태계, 풍부한 플러그인 | 설정 복잡, UI 구식 |
| **Zabbix** | Pull+Push / 오픈소스 | 올인원 솔루션, 에이전트 지원 | 무겁고 복잡, 학습 곡선 높음 |
| **Datadog** | Push / 상용 | 사용 편리, SaaS, 통합 플랫폼 | 비용 높음, 벤더 락인 |
| **ELK Stack** | Push / 오픈소스 | 로그 분석 강력, 풀텍스트 검색 | 리소스 소모 큼, 메트릭 약함 |
| **New Relic** | Push / 상용(프리미엄) | APM 특화, 코드 수준 분석 | 가격, 커스터마이징 제한 |
